# Como Usar Este Repositório — Projeto Trail

Este documento explica **como o repositório do Projeto Trail está organizado**
e **como os alunos devem utilizá‑lo ao longo da mentoria**.

📌 Este repositório é a **fonte oficial de documentação do projeto**.

---

## 🎯 Objetivo deste Repositório

Este repositório existe para:

- Centralizar toda a documentação do projeto
- Registrar decisões e aprendizados
- Servir como memória oficial da mentoria
- Simular a organização de um projeto profissional real

⚠️ Este repositório **não contém código**.  
O código do projeto será mantido em repositórios separados.

---

## 📄 Fonte Única da Verdade

> O **Documento de Briefing / Discovery** é a fonte única da verdade do projeto.

Regras importantes:
- O briefing **define o problema, visão e objetivos**
- Os encontros **interpretam e recortam o briefing**
- O briefing **não deve ser alterado**, apenas referenciado

📌 Todo conteúdo deste repositório deve:
- Derivar do briefing
- Ou justificar claramente qualquer adaptação

---

## 📁 Estrutura de Pastas

A estrutura do repositório segue um padrão fixo:

```text
/
├── README.md
├── regras/
├── discovery/
├── encontros/
├── atividades/
├── entregas/
└── referencias/
```

---

### 📂 regras/
Contém documentos obrigatórios de governança.
Exemplos:
- Governança e LGPD
- Como usar este repositório
- Regras de colaboração


### 📂 discovery/
Contém o Documento de Briefing / Discovery.
📌 Este conteúdo é base conceitual do projeto e não muda a cada encontro.

### 📂 encontros/
Contém um arquivo Markdown para cada encontro da mentoria.
Padrão:
- encontros/XX-nome-do-encontro.md

Exemplo:
- encontros/02-problema-e-requisitos.md

Cada arquivo de encontro define:

- Objetivo
- Conteúdo
- Atividades
- Critérios de aceite


### 📂 atividades/
Contém materiais transversais, reutilizados ao longo do projeto.
Exemplos:
- Checklist semanal
- Desafios técnicos
- Orientações gerais


### 📂 entregas/
Contém os artefatos produzidos pelos alunos entre encontros.
Padrão recomendado:
- entregas/encontro-XX-insumos.md

📌 As entregas devem:
- Estar em Markdown
- Ser claras e autoexplicativas
- Poder ser usadas no próximo encontro


### 📂 referencias/
Material de apoio complementar:
- Boas práticas
- Links
- Diagramas conceituais
- Conteúdo opcional


### 📝 Padrão de Escrita (Markdown)
Todos os arquivos devem:
- Estar em formato .md
- Usar títulos claros (#, ##, ###)
- Usar listas e tabelas quando necessário
- Ser compreensíveis sem explicação verbal

📌 Escreva sempre pensando:
“Alguém que não participou do encontro consegue entender isso?”


### ✅ Boas Práticas de Uso
- Versione tudo que for decisão
- Prefira clareza a volume
- Justifique escolhas importantes
- Use linguagem simples e objetiva
- Atualize documentos quando necessário (com histórico)


### 🚫 O que NÃO fazer
- Não criar documentos fora do repositório
- Não manter decisões apenas “em conversa”
- Não subir arquivos com dados sensíveis
- Não misturar código com documentação
- Não criar estruturas de pastas paralelas


### 🧭 Fluxo de Trabalho Esperado
- Encontro acontece
- Atividades são definidas
- Alunos produzem entregas
- Entregas são versionadas
- Próximo encontro utiliza esses artefatos

📌 Se não está no repositório, não existe oficialmente.

### 🎓 Aprendizado Esperado
Ao final da mentoria, o aluno terá vivenciado:
- Organização de documentação profissional
- Uso de repositório como fonte de verdade
- Escrita técnica clara
- Disciplina de versionamento
- Governança básica de projeto


#### Lembrete final:
*Bons projetos não são organizados depois.*
*Eles nascem organizados.*
