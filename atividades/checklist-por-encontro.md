# Checklist por Encontro — Projeto Trail

Este checklist define o **ritmo padrão de trabalho semanal** do Projeto Trail.

📌 Ele se aplica a **todos os encontros da mentoria**, independentemente do tema técnico.

O objetivo é:
- Garantir consistência
- Criar autonomia no time
- Simular a cadência de projetos profissionais reais

---

## 🧭 Regra Fundamental do Projeto

> O **Documento de Briefing / Discovery** é a **fonte única da verdade** do projeto.

Todo artefato, decisão ou entrega deve:
- Derivar do briefing  
- Ou justificar claramente qualquer adaptação feita

---

## 📆 Estrutura de uma Semana de Mentoria

Cada ciclo semanal segue este fluxo:

1. Encontro síncrono (mentoria)
2. Definição clara de entregáveis
3. Execução assíncrona pelo time
4. Entrega em Markdown
5. Uso direto da entrega no próximo encontro

📌 **Não existe encontro isolado.**  
Cada semana depende da anterior.

---

## ✅ Checklist — ANTES do Encontro

Antes de cada encontro, o time deve:

- [ ] Ler o **Documento de Briefing**
- [ ] Revisar os artefatos do encontro anterior
- [ ] Entender claramente:
  - Qual é o problema sendo tratado
  - Qual é o objetivo do encontro atual
- [ ] Trazer dúvidas registradas (não apenas mentais)

---

## ✅ Checklist — DURANTE o Encontro

Durante o encontro, o time deve:

- [ ] Confirmar entendimento do problema
- [ ] Questionar decisões quando necessário
- [ ] Registrar decisões importantes
- [ ] Identificar claramente:
  - O que é MVP
  - O que fica para depois
- [ ] Sair do encontro sabendo:
  > “O que preciso entregar até a próxima semana?”

---

## ✅ Checklist — APÓS o Encontro (ENTRE ENCONTROS)

Após cada encontro, o time deve:

- [ ] Produzir os **artefatos solicitados**
- [ ] Utilizar **Markdown (.md)** como formato padrão
- [ ] Garantir que as entregas:
  - São claras
  - São objetivas
  - Estão alinhadas ao briefing
- [ ] Nomear arquivos conforme padrão definido
- [ ] Versionar o conteúdo no repositório

---

## 📦 Padrão de Entregas

- Um arquivo `.md` por encontro (salvo exceções)
- Nome sugerido: `atividade-XX-descricao.md`
- Conteúdo deve incluir:
  - Contexto
  - Decisões tomadas
  - Justificativas (quando aplicável)

---

## ✅ Critérios Gerais de Qualidade

Uma entrega é considerada adequada se:

- [ ] Pode ser entendida sem explicação verbal
- [ ] Possui relação clara com o briefing
- [ ] Não contém dados sensíveis ou pessoais
- [ ] Pode ser reutilizada no próximo encontro

---

## 🚫 O que EVITAR

- Entregas vagas ou genéricas
- Conteúdo não versionado
- Decisões apenas verbais
- Funcionalidades sem problema associado

---

## 🎯 Responsabilidade do Time

> O mentor orienta.  
> O time constrói.

Cada encontro pressupõe que:
- O time é responsável pela execução
- O repositório é a memória do projeto
- O aprendizado acontece na prática

---

## 🔗 Uso nos Próximos Encontros

Este checklist será utilizado:
- Como referência fixa da mentoria
- Como critério de acompanhamento
- Como base para feedbacks

📌 Se o checklist não for seguido, o projeto perde ritmo.

---

> **Lembrete final:**  
> Projetos profissionais não falham por tecnologia.  
> Falham por falta de clareza, disciplina e registro.
