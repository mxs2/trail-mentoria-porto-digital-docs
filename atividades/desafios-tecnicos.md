# Desafios Técnicos — Projeto Trail

Este documento define **o que são desafios técnicos**,  
**como eles devem ser propostos**,  
e **como devem ser entregues e avaliados** no contexto do Projeto Trail.

📌 Este padrão se aplica a **todos os desafios do projeto**, independentemente da tecnologia.

---

## 🎯 Objetivo dos Desafios Técnicos

Os desafios técnicos existem para:

- Transformar teoria em prática
- Validar aprendizado real
- Simular problemas do mundo profissional
- Gerar insumos mensuráveis para mentoria e avaliação

📌 Um desafio técnico **não é um exercício escolar**.  
Ele representa uma **situação real de projeto**.

---

## 📄 Fonte Única da Verdade

> O **Documento de Briefing / Discovery** é a fonte única da verdade.

Regras:
- Todo desafio deve estar relacionado a um problema descrito no briefing
- Nenhum desafio existe “por tecnologia”
- Tecnologia é meio, não fim

📌 Se um desafio não resolve um problema do briefing, ele **não deve existir**.

---

## 🧩 O que é um Desafio Técnico

Um desafio técnico é uma atividade prática que:

- Resolve um problema específico do projeto
- Gera um artefato concreto
- Pode ser analisado, discutido e melhorado
- Produz aprendizado verificável

Exemplos:
- Definir um modelo de dados
- Criar contratos de API
- Propor arquitetura de autenticação
- Definir métricas e KPIs

---

## 🚫 O que NÃO é um Desafio Técnico

Não são considerados desafios técnicos:

- Tutoriais genéricos
- Atividades sem relação com o projeto
- Exercícios sem contexto
- Copiar código ou conteúdo pronto
- Atividades sem entrega verificável

---

## 🏗️ Estrutura Padrão de um Desafio

Todo desafio técnico deve conter:

1. **Contexto**
2. **Problema a ser resolvido**
3. **Objetivo**
4. **Instruções**
5. **Entrega esperada**
6. **Critério de aceite**

📌 Sem esses itens, o desafio é considerado incompleto.

---

## 📝 Exemplo de Desafio Técnico

```text
Contexto:
O Projeto Trail precisa acompanhar o progresso dos estudantes.

Problema:
Hoje não existe uma forma padronizada de visualizar progresso e métricas.

Objetivo:
Definir quais métricas representam progresso real.

Entrega esperada:
Documento Markdown com definição de KPIs.

Critério de aceite:
As métricas estão alinhadas ao briefing e são mensuráveis.
```


## 📦 Formato das Entregas

- Todas as entregas devem ser feitas em Markdown (.md)
- Um arquivo por desafio (salvo exceções)
- Nome sugerido:
  - entregas/encontro-XX-desafio-YY.md



Exemplo:
- entregas/encontro-04-desafio-01.md


## ✅ Critérios Gerais de Avaliação
Um desafio será considerado bem executado se:
- Está claramente relacionado ao briefing
- Resolve um problema real
- Possui clareza e objetividade
- Pode ser entendido sem explicação verbal
- Gera insumos para próximos encontros

📌 Avaliação considera qualidade, não volume.

## 🧠 Papel do Mentor nos Desafios
O mentor:
- Orienta o raciocínio
- Questiona decisões
- Ajuda a refinar soluções
- Não entrega respostas prontas

📌 O erro faz parte do aprendizado, desde que seja analisado.

## 🧭 Evolução dos Desafios ao Longo do Projeto
Os desafios:
- Evoluem em complexidade
- Se conectam entre si
- Constroem o produto gradualmente

📌 Um desafio bem feito vira base para o próximo.

## ⚠️ Boas Práticas Importantes

- Prefira soluções simples e justificadas
- Documente decisões importantes
- Explique o “por quê”, não apenas o “como”
- Use linguagem técnica clara
- Evite excesso de abstração


## 🚫 Erros Comuns a Evitar

- Criar desafios grandes demais
- Misturar múltiplos problemas em um desafio
- Ignorar o briefing
- Entregar algo que não será usado depois


## 🎓 Aprendizado Esperado
Ao final da mentoria, o aluno terá aprendido a:

- Resolver problemas reais com tecnologia
- Documentar soluções técnicas
- Defender decisões
- Trabalhar com critérios de aceite
- Evoluir soluções de forma incremental



## 📝 Lembrete final
*Bons profissionais não se destacam por saber mais tecnologias.
Se destacam por resolver melhor os problemas certos.*
